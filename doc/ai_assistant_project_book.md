# HoloCubic AI 桌面助手项目书

## 1. 项目概述

本项目目标是构建一个由 `macOS + ESP32-S3` 组成的桌面 AI 助手系统。

系统角色分工如下：

- `macOS` 负责运行语音识别、对话编排和大语言模型
- `ESP32-S3-PICO-1-N8R8` 负责桌面摆件形态的外设侧能力
- 摆件主要承担：
  - 麦克风原始音频采集
  - 屏幕状态显示
  - Wi-Fi 联网
  - 固件 OTA 更新接收

本项目不在 `ESP32` 侧进行大模型推理，也不在 `ESP32` 侧承担完整语音识别任务。`ESP32` 只负责采集和显示，识别与生成全部由 `Mac` 完成。

## 2. 项目目标

本项目分为两个目标层级。

### 2.1 第一阶段目标

完成一个可稳定运行的桌面 AI 助手原型，满足以下要求：

- `ESP32` 持续监听麦克风
- 原始音频低延迟传输到 `Mac`
- `Mac` 完成语音识别与大模型对话
- `ESP32` 屏幕显示像素风表情动画和字幕文本
- `ESP32` 始终接入 Wi-Fi
- `Mac` 可通过网络向 `ESP32` 推送并更新固件

### 2.2 第二阶段目标

将原型扩展为长期桌面常驻系统，满足以下要求：

- 自动重连
- OTA 校验与回滚
- 本地网页控制台
- 常驻后台服务
- 调试与运行日志
- 更稳定的 UI 状态管理

## 3. 已确认需求

以下需求已经明确，作为当前项目冻结输入。

### 3.1 硬件侧

- 主控：`ESP32-S3-PICO-1-N8R8`
- 显示屏：`ST7789V`
- 分辨率：`240 x 40`
- 接口：`SPI`
- 麦克风：板载数字麦克风，仅做采集
- 不需要声音播放
- 无物理按键

### 3.2 软件与交互侧

- 系统需要“一直监听”
- `ESP32` 不做本地语音识别
- `ESP32` 只采集原始音频并发送给 `Mac`
- `Mac` 侧需要：
  - 普通桌面 App
  - 后台服务
  - 本地网页控制台
- 显示风格采用“像素风”
- 屏幕显示内容包括：
  - 表情动画
  - 字幕文本

### 3.3 联网与传输侧

- `ESP32` 无论如何都要连接 Wi-Fi
- 实际通信传输优先级为：
  1. `USB CDC`
  2. `Wi-Fi`
  3. `BLE`

### 3.4 固件更新侧

- OTA 由 `Mac` 主动推送到 `ESP32`
- 固件更新必须带校验机制

### 3.5 性能目标

- 除去模型推理时间，链路延迟应尽量低

## 4. 非目标范围

当前阶段明确不做以下内容：

- `ESP32` 端大语言模型推理
- `ESP32` 端完整语音识别
- `ESP32` 端声音播放
- 依赖按键进行主要交互
- 依赖 BLE 传输持续音频流

## 5. 总体架构

建议采用“Mac 中心化”的双端架构。

### 5.1 ESP32 端职责

- 持续采集麦克风原始音频
- 管理显示屏 UI
- 建立与 `Mac` 的多链路通信
- 保持 Wi-Fi 在线
- 接收 OTA 固件
- 上报设备状态

### 5.2 macOS 端职责

- 接收音频流
- 执行语音前处理
- 执行语音识别
- 执行对话编排与大模型调用
- 生成字幕与表情状态
- 将显示内容下发到 `ESP32`
- 统一管理 OTA、配置、日志与调试界面

## 6. 通信设计

### 6.1 连接原则

系统中 Wi-Fi 是“必连链路”，原因如下：

- OTA 依赖网络
- 本地网页控制台依赖网络
- 后台服务发现与管理依赖网络
- 桌面常驻场景中，无线状态同步更稳定

但“业务数据的实时传输”仍按优先级选择：

1. `USB CDC`
2. `Wi-Fi`
3. `BLE`

### 6.2 各链路用途建议

#### USB CDC

优先承担：

- 原始音频流传输
- 调试日志
- 开发阶段控制命令

优点：

- 延迟低
- 稳定
- 无需额外配网
- 开发调试效率高

缺点：

- 有线约束明显

#### Wi-Fi

优先承担：

- 在线状态同步
- 字幕/动画指令下发
- OTA 固件推送
- 本地控制台访问
- 无线模式下的音频传输

优点：

- 适合长期桌面常驻
- 吞吐能力足够
- 适合 OTA 和网页控制台

缺点：

- 首次配网需要设计
- 无线网络抖动需要处理

#### BLE

建议只承担：

- 设备发现
- 首次配网
- 极简控制命令

不建议承担：

- 持续原始音频流
- 高频字幕刷新
- OTA 主链路

### 6.3 链路选择策略

建议系统启动后按以下规则工作：

1. `ESP32` 开机立即连接已保存 Wi-Fi
2. 同时检测 USB 是否与 `Mac` 建立 CDC 通道
3. 如果 USB 可用，则音频主通道切到 USB
4. 如果 USB 不可用，则退回 Wi-Fi 音频传输
5. BLE 仅在配网或异常恢复时启用

## 7. 音频方案

### 7.1 音频职责边界

音频识别任务只要求在 `Mac` 侧完成，因此 `ESP32` 端只负责：

- 连续采集
- 打包
- 传输

`ESP32` 不负责：

- VAD
- 唤醒词
- ASR
- NLU

### 7.2 建议音频格式

建议第一版直接固定为：

- `PCM`
- `16 kHz`
- `16-bit`
- `Mono`

原因：

- 足够完成语音识别任务
- 带宽较低
- 工程实现简单
- 与大部分 `Mac` 侧 ASR 流程兼容

该格式下原始数据吞吐约为：

- `16000 x 2 = 32000 bytes/s`
- 即约 `32 KB/s`

这个量级对 `USB CDC` 和 `Wi-Fi` 都是可接受的。

### 7.3 传输粒度建议

建议以固定音频帧发送：

- 每帧 `20ms`
- 每帧 `320 samples`
- 每帧 `640 bytes`

这样便于：

- 低延迟流式处理
- Mac 侧缓冲与重组
- 后续接入 VAD / ASR

## 8. 显示系统设计

### 8.1 屏幕特征

屏幕为 `ST7789V SPI 240x40`，高度非常有限，因此不适合显示多行长文本。

更适合的 UI 形式是“窄横条状态屏”。

### 8.2 推荐界面布局

建议将屏幕划分为两个区域：

- 左侧：像素风表情动画
- 右侧：单行字幕或状态文本

### 8.3 推荐显示状态

建议至少定义以下 UI 状态：

- `BOOT`
- `WIFI_CONNECTING`
- `IDLE_LISTENING`
- `RECORDING`
- `UPLOADING_AUDIO`
- `TRANSCRIBING`
- `THINKING`
- `DISPLAYING_REPLY`
- `OTA_UPDATING`
- `ERROR`

### 8.4 像素风动画方向

建议像素风以低帧率、小尺寸、强状态区分为原则。

适合的表现方式：

- 眨眼
- 嘴型变化
- 呼吸灯式脸部表情
- 波形跳动
- 错误表情
- OTA 进度动画

不建议第一版就做过度复杂的逐帧全屏动画，以免影响刷新效率和开发节奏。

## 9. OTA 设计

### 9.1 OTA 目标

OTA 需要支持：

- `Mac` 主动推送固件
- 固件完整性校验
- 版本管理
- 更新失败保护

### 9.2 OTA 推荐流程

建议采用以下流程：

1. `Mac` 请求 `ESP32` 进入 OTA 模式
2. `ESP32` 返回设备信息
   - 当前版本
   - 芯片型号
   - 固件分区信息
   - 可用空间
3. `Mac` 发送固件元信息
   - 固件版本
   - 固件大小
   - `SHA-256`
4. `ESP32` 校验元信息后进入接收状态
5. `Mac` 分块发送固件
6. `ESP32` 写入 OTA 分区
7. `ESP32` 完成整包校验
8. 校验成功后标记新分区可启动
9. 重启切换新固件

### 9.3 OTA 安全建议

第一版最低要求：

- `SHA-256` 完整性校验
- 固件版本号检查
- 固件大小校验
- 分块序号校验

桌面常驻版建议增加：

- 固件签名校验
- 安全版本号限制
- 启动成功确认机制
- 回滚机制

## 10. macOS 端软件形态

当前需求明确要求同时具备三层形态。

### 10.1 桌面 App

- 用户可视化交互

- 音频状态显示
- 固件升级入口

### 10.2 后台服务

- 设备发现

- 音频收发
- ASR/LLM 调度
- 日志

### 10.3 本地网页控制台

负责：

- 设备列表

## 11. 技术路线建议

由于 `Mac` 侧技术栈尚未确定，建议先按“架构约束”而不是“框架偏好”选型。

### 11.1 Mac 端推荐分层

- `Core Service`
  - 常驻进程
  - 负责连接、音频、识别、模型、状态同步
- `Desktop App`
  - 仅作为 UI 外壳
- `Local Web Console`
  - 读取后台服务状态并发指令

### 11.2 Mac 端能力要求

无论最终使用 `Swift`、`Python` 或混合方案，都需要满足：

- 音频流式接收
- 流式 ASR
- 本地 LLM 调用
- WebSocket / HTTP 能力
- OTA 上传能力
- 多链路设备管理能力

## 12. 性能目标

### 12.1 低延迟目标

在不计算模型推理时间的情况下，系统应尽量减少：

- `ESP32` 采样缓冲延迟
- 音频分帧等待
- 链路发送等待
- Mac 侧接收缓冲等待
- 显示状态下发等待

### 12.2 设计原则

建议遵守以下原则：

- 音频帧小而固定
- 显示指令轻量化
- 避免 BLE 承担高频数据
- 状态机优先于复杂脚本控制
- Mac 侧做统一缓冲和调度

## 13. 可靠性要求

因为项目既要做开发验证，也要面向长期常驻，所以必须从一开始就保留可靠性设计。

### 13.1 必做项

- 断线重连
- USB / Wi-Fi 主通道切换
- OTA 校验
- 日志
- 异常状态显示
- 启动自检

### 13.2 后续增强项

- 固件回滚
- 配网恢复流程
- 心跳保活
- 本地缓存最近状态
- 局域网自动发现

## 14. 风险与约束

### 14.1 无按键约束

由于系统没有按键，以下操作必须通过软件实现：

- 配网恢复
- 强制进入 OTA 模式
- 恢复出厂设置
- 重启

因此建议至少预留以下软件入口：

- 串口命令
- 本地网页控制台
- BLE 配网模式

### 14.2 窄屏约束

`240x40` 屏幕高度很小，因此：

- 文本必须短
- 长文本必须滚动
- UI 不能依赖多行布局
- 动画与字幕要抢占同一横向空间

### 14.3 一直监听约束

一直监听意味着：

- 音频链路必须长期稳定
- Mac 后台服务必须常驻
- 音频帧不能无限缓存
- 必须有连接异常恢复方案

## 15. 第一版实现建议

建议第一版按以下范围控制复杂度。

### 15.1 ESP 第一版

- 连 Wi-Fi
- 建 USB CDC
- 采集 `16k/16bit/mono PCM`
- 通过 `USB/Wi-Fi` 发音频帧
- 显示基础像素风状态动画
- 显示单行字幕
- 提供 OTA 接口

### 15.2 Mac 第一版

- 后台服务接收音频
- 流式语音识别
- 调用本地 LLM
- 输出状态指令给 `ESP`
- 提供简单桌面 App
- 提供基础网页控制台

### 15.3 暂缓项

- BLE 高频业务传输
- 复杂多页面 UI
- 本地唤醒词
- 多设备管理
- 高级动画系统

## 16. 里程碑计划

### 里程碑 M1：硬件联调

- 屏幕点亮
- 音频采集稳定
- Wi-Fi 常驻
- USB CDC 通信可用

### 里程碑 M2：链路打通

- 音频通过 USB 发到 Mac
- Mac 返回字幕文本
- 屏幕显示字幕和基础表情

### 里程碑 M3：AI 闭环

- Mac 完成 ASR + LLM
- 摆件正确显示状态变化
- 整链路延迟可接受

### 里程碑 M4：OTA 与控制台

- Wi-Fi OTA 推送
- 校验成功后可稳定升级
- 本地网页控制台可用

### 里程碑 M5：桌面常驻版

- 自动重连
- 异常恢复
- 版本管理
- 长时间运行稳定性验证

## 17. 当前结论

基于目前需求，项目的推荐方向已经明确：

- `ESP32` 作为外设终端，不做识别，只做采集、显示、联网和 OTA
- `Mac` 作为中心控制节点，承担识别、模型、管理与更新
- `Wi-Fi` 必连，`USB` 优先传输，`BLE` 只做补充链路
- UI 采用像素风窄横条设计
- OTA 采用 `Mac -> ESP32` 的推送模式，并带校验

该方案与当前硬件条件匹配，能够同时覆盖“快速原型验证”和“长期桌面常驻”两个目标。
